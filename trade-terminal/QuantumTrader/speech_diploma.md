### ~10 минут | Редактируй под себя

## Слайд 1 — Титульный (30 сек)

Добрый день. Меня зовут Евгений, тема моей дипломной работы — **QuantumTrader Pro**, торговый терминал реального времени, разработанный на языке C++ с использованием фреймворка Qt и аппаратного рендеринга через OpenGL.

Терминал подключается к криптобирже Bybit по WebSocket и отображает рыночные данные в реальном времени.

---

## Слайд 2 — Актуальность (1 мин)

Почему это актуально?

Существующие браузерные торговые терминалы имеют три принципиальных ограничения.

Во-первых, **скорость обработки данных**. Биржа Bybit отправляет обновления тысячи раз в секунду. Браузерные решения на JavaScript не успевают обрабатывать такой поток без потерь и задержек.

Во-вторых, **рендеринг графики**. Когда на экране одновременно работают график свечей, стакан заявок и лента сделок — нужна аппаратная графика. CSS и Canvas с этим не справляются.

В-третьих, **расширяемость**. Профессиональные трейдеры хотят писать собственные индикаторы, не дожидаясь обновлений от разработчика.

**Решение** — нативный десктопный терминал на C++, где всё это решается на уровне архитектуры.

---

## Слайд 3 — Архитектура (1.5 мин)

Система построена в три слоя.

**Первый слой — источник данных.** Это Bybit WebSocket API. Биржа в реальном времени отправляет три типа сообщений: kline (свечи), orderbook (стакан) и publicTrade (лента сделок).

**Второй слой — бизнес-логика.** Здесь четыре ключевых компонента. BybitConnector управляет WebSocket-соединением. ByBitParser разбирает входящий JSON. MarketDataManager управляет подписками — какой виджет слушает какой символ. И EventBus — это шина событий на Qt-сигналах, которая связывает все компоненты между собой, не создавая прямых зависимостей.

**Третий слой — интерфейс.** ChartContainer, OrderBookContainer и TimeAndSalesContainer — каждый отображает свой тип данных. WindowManager на базе Qt ADS управляет расположением всех окон.

Главное преимущество такой архитектуры — модули не знают друг о друге. Добавить новый виджет или новую биржу можно, не трогая существующий код.

## Слайд 4 — Технологический стек (1 мин)

Теперь об инструментах.

**C++17** — основной язык. Даёт максимальную производительность, управление памятью через RAII и zero-cost абстракции.

**Qt 6** — UI-фреймворк. Система сигналов и слотов идеально подходит для событийной архитектуры. Qt ADS добавляет полноценные dock-панели, как в профессиональных IDE.

**OpenGL** — аппаратный рендеринг через QOpenGLWidget. Графика исполняется на GPU, поэтому тысячи свечей рисуются без падения FPS.

**simdjson** — самый быстрый JSON-парсер, использует SIMD-инструкции процессора. Биржевые сообщения разбираются за микросекунды.

**WebSocket** — асинхронный канал для получения данных в реальном времени без постоянных HTTP-запросов.

**CMake** с GLOB_RECURSE — система сборки автоматически подхватывает новые файлы, что упрощает разработку.

## Слайд 5 — Реализованные модули (2 мин)

В рамках работы реализованы три функциональных модуля.

**График свечей.** Построен на QOpenGLWidget — GPU рисует каждую свечу напрямую. Поддерживается загрузка исторических данных через REST API и live-обновление через WebSocket. Пользователь может масштабировать график колёсиком мыши и панорамировать. Поддерживаются все таймфреймы Bybit — от одной минуты до месяца.

**Стакан заявок, Order Book.** Получает и обрабатывает как полный снимок, так и инкрементальные delta-обновления. Реализованы три независимо перетаскиваемых колонки — Price, Size, Total. Пользователь мышью регулирует ширину каждой. Справа от колонок отображается индикатор объёма в виде цветного бара. Ключевая особенность: при любом сужении окна все три колонки остаются видимы.

**Лента сделок, Time and Sales.** Получает данные через топик publicTrade Bybit. Каждая строка — реальная сделка с биржи. Зелёный фон — покупка, красный — продажа. Реализован авто-скролл, ручная прокрутка колёсиком и буфер на 500 последних тиков.

## Слайд 6 — Производительность (1 мин)

Три ключевых показателя производительности.

Парсинг JSON через simdjson занимает **меньше одной микросекунды** на сообщение — это примерно в 10 раз быстрее стандартных парсеров за счёт SIMD-инструкций.

Рендеринг интерфейса работает на стабильных **60 кадрах в секунду** — OpenGL исполняется на GPU и не загружает основной поток.

Задержка данных от биржи через WebSocket составляет **менее 50 миллисекунд** — данные приходят практически мгновенно.

Архитектурно я применил три решения, которые этому способствуют: EventBus устраняет лишние зависимости, абсолютные пиксели колонок исключают пересчёт на каждый фрейм, паттерн Container/Widget разделяет логику подписки и логику рендеринга.

## Слайд 7 — Перспективы (1 мин)

Система спроектирована с расчётом на развитие.

Главная перспектива — **Lua-скриптинг индикаторов**. Трейдер пишет свой индикатор на языке Lua, не зная C++. C++-хост предоставляет API: доступ к данным свечей, функции EMA, SMA, возможность рисовать линии на графике. Скрипт загружается в рантайме без перекомпиляции терминала. Архитектура EventBus уже готова к такому расширению.

Из дополнительных направлений: алерты по цене с уведомлениями, поддержка нескольких бирж через единый интерфейс IConnector, и Watchlist — список инструментов с live-ценами.

## Возможные вопросы и ответы

**— Почему C++, а не Python или Java?**
C++ даёт прямой контроль над памятью и производительность без garbage collector. Для торгового терминала, где каждая миллисекунда важна, это принципиально.

**— Как обеспечивается потокобезопасность при работе с WebSocket?**
WebSocket-соединения работают в отдельных потоках пула. EventBus использует Qt-сигналы с Qt::QueuedConnection — это автоматически маршрутизирует вызовы в нужный поток.

**— Почему выбрана именно Bybit, а не другая биржа?**
Bybit имеет подробную документацию WebSocket API и позволяет работать без аутентификации для публичных данных. Архитектура через IConnector позволит добавить другие биржи в будущем.

**— Что такое delta-обновления в стакане заявок?**
Биржа сначала присылает полный снимок стакана (snapshot), а затем только изменения (delta) — уровни, которые появились или исчезли. Это экономит трафик и ускоряет обновление.

**— Тестировалась ли система под нагрузкой?**
Система тестировалась на реальных данных Bybit в режиме реального времени. При одновременной работе графика, стакана и ленты сделок по паре BTCUSDT рендеринг остаётся стабильным на 60 FPS.
